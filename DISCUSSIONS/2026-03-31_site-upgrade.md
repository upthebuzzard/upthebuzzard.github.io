# Site Upgrade Discussion

## 2026-03-31 — Initial Assessment

**What it is**: A Jekyll-based personal storytelling site at `stories.upthebuzzard.com`, hosted on GitHub Pages. Contains children's stories, flash fiction, haiku, game design notes, and more across 11 custom collections (~100 content files).

### Strengths

- **Well-organized collections system** — 11 collections with rich metadata (sequence ordering, background images, per-collection RSS feeds, credits)
- **GitHub Pages compatible** — No custom Ruby plugins; sticks to jekyll-feed, jekyll-sitemap, jekyll-redirect-from
- **Creative features** — N-gram Markov chain text generator for auto-mashup content, custom `<radio-script>` XML elements for dialogue formatting, porthole background effect
- **Good content structure** — Consistent frontmatter, prev/next navigation, story details footers with licensing (CC BY-NC-SA 4.0)
- **Client-side search** with permutation-based matching
- **Responsive design** with breakpoints at 600px/800px

| **Analytics** | Google Analytics with a `UA-` tracking ID (Universal Analytics was sunset July 2024; should migrate to GA4) |

### Content Stats

~100 content files across collections, most active being **predicting_the_present** (23 posts), **emus_all_the_way_down** (16), **moose_and_goose_stories** (14), and **jekyll_notes** (12). Recent activity includes a new story and game notes updates.

### Summary

A well-crafted, mature personal site with thoughtful architecture. The main areas for modernization would be updating the Google Analytics integration, switching the config URL to HTTPS, compressing oversized images, and cleaning up the stale `test.txt`. The Jekyll setup itself is solid and plays nicely within GitHub Pages constraints.

## 2026-03-31 — Getting the Site to Build and Run Locally

### Starting state

- **Ruby 3.4.1** installed at `~/.rubies/ruby-3.4.1/` (via `ruby-install`)
- **Bundler 1.17.2** — the only version installed, far too old for Ruby 3.4
- **Gemfile.lock** — generated by Bundler 1.17.2, pinning all gem versions to that era
- **chruby** — referenced in shell config but not installed (harmless warning, not a blocker)

### Problem 1: Bundler 1.17.2 incompatible with Ruby 3.4

Bundler 1.17.2 calls `String#untaint`, which was removed in Ruby 3.4. Every `bundle` command crashed immediately.

**Fix:**
1. Installed modern Bundler: `gem install bundler` (got 4.0.9)
2. Removed old Bundler: `gem uninstall bundler -v 1.17.2`
3. Deleted stale `Gemfile.lock` — it was pinned to `BUNDLED WITH 1.17.2`, and Bundler 4.0.9 kept re-downloading and delegating to 1.17.2, which then crashed. Deleting the lockfile let the new Bundler generate a fresh one.
4. Ran `bundle install` — successfully resolved and installed 95 gems.

### Problem 2: Ruby 3.4 removed stdlib gems that Jekyll/Liquid depend on

Ruby 3.4 removed several libraries from the default gem set. Jekyll 3.9.2 and its dependencies (Liquid 4.0.3, etc.) still `require` them, causing `LoadError` at build time.

**Fix:** Added these to the `Gemfile`:

```ruby
# Required for Ruby 3.1+ / 3.4+ (removed from default gems)
gem "webrick"      # needed for `jekyll serve` (removed in Ruby 3.1)
gem "csv"          # needed by Jekyll 3.9.2 (removed in Ruby 3.4)
gem "bigdecimal"   # needed by Liquid 4.0.3 (removed in Ruby 3.4)
gem "base64"       # needed by various deps (removed in Ruby 3.4)
```

### Problem 3: Liquid 4.0.3 uses removed taint methods

The `github-pages` gem (~226) pins Liquid to 4.0.3, which calls `String#untaint`, `String#tainted?`, and related methods throughout its template rendering code. These were fully removed in Ruby 3.4 (deprecated since 2.7, no-op in 3.2, gone in 3.4).

This is a fundamental incompatibility — Liquid 4.0.3 cannot run on Ruby 3.4 without patching, and we can't upgrade Liquid because `github-pages` pins it.

**Fix:** Created `_plugins/ruby34_compat.rb` — a monkey-patch that restores all taint-related methods as harmless no-ops:

```ruby
if RUBY_VERSION >= "3.4"
  module TaintShim
    def taint; self; end
    def untaint; self; end
    def tainted?; false; end
    def trust; self; end
    def untrust; self; end
    def trusted?; true; end
  end
  Object.prepend(TaintShim)
end
```

**Key detail:** This shim must be loaded *before* Jekyll and Liquid start. Jekyll's `_plugins/` directory is loaded too late (after Liquid has already crashed during rendering). The fix is to use `RUBYOPT` to require the file at Ruby startup:

```bash
RUBYOPT="-r ./_plugins/ruby34_compat.rb" bundle exec jekyll serve
```

### Wrapper script: `serve.sh`

To avoid remembering the `RUBYOPT` flag, created `serve.sh`:

```bash
#!/bin/bash
export RUBYOPT="-r $(dirname "$0")/_plugins/ruby34_compat.rb"
bundle exec jekyll serve "$@"
```

Usage: `./serve.sh` (accepts any Jekyll flags, e.g. `./serve.sh --drafts`)

### Config changes

Added to `_config.yml` exclude list:

- `CLAUDE.md` — Claude Code project instructions, not site content

### Result

Site builds in ~1.5 seconds and serves locally at `http://127.0.0.1:4000/`. No changes needed to any content or template files.

### Notes for the future

- The taint shim is only needed because `github-pages` pins old Liquid. If GitHub Pages ever updates to Liquid 5+, the shim can be removed.
- An alternative to the shim would be downgrading to Ruby 3.3.x (where taint methods are no-ops rather than missing). Ruby 3.3 was attempted via `ruby-install` but the compilation timed out.
- The `_plugins/` directory and `serve.sh` are for local builds only. GitHub Pages ignores `_plugins/` (safe mode) and doesn't use `serve.sh`, so the live site is unaffected.
